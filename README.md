# Cortex Lite

## What it is

A portfolio web app for PC gamers: Steam-connected game library, manual session tracking, AI-assisted graphics-settings optimizer (deterministic rule-based engine + Gemini API for prose explanations only), and Stripe freemium tier. Built to demonstrate the backend, cloud, and AI-integration skills.

**Stack:** Laravel 13 ┬╖ PHP 8.4 ┬╖ MySQL 8 ┬╖ Redis 7 ┬╖ React 19 (Vite) ┬╖ Stripe ┬╖ Docker ┬╖ AWS ┬╖ Gemini API ┬╖ Steam Web API ┬╖ PCGamingWiki API

## How to run it locally

### Prerequisites (any OS)

Install these first тАФ everything else runs inside Docker, so you do **not** need PHP, Composer, or MySQL on the host.

- **Docker Desktop** (or Docker Engine + Compose v2). Verify with `docker compose version`.
- **Node.js 20+** and npm тАФ the React dev server runs on the host. Verify with `node -v`.
- **git**.
- **GNU Make** тАФ used by the `make` shortcuts below.
  - macOS: preinstalled (or `xcode-select --install`). Linux: `apt install make`.
  - Windows: comes with **Git Bash** if you tick "Git Bash" tooling, or install via `choco install make` / `winget install GnuWin32.Make`. Run all `make` commands from a **Git Bash** shell, not PowerShell/CMD.
  - No make? Every `make X` maps to a plain `docker compose тАж` command тАФ see [Running without make](#running-without-make).

### First-time setup (fresh clone, any machine)

```bash
git clone <repo-url> Cortex-Mini
cd Cortex-Mini

cp .env.example .env        # create your local env file (never committed)

make up                     # build + start all 6 services (first run pulls images, ~2тАУ5 min)
make install                # composer install + php artisan key:generate + migrate (inside the app container)
make artisan CMD="db:seed"  # seed hardware tables + optimizer anchors
make artisan CMD="db:seed --class=DemoAccountSeeder"  # seed demo account

make client-install         # cd client && npm install
make client-dev             # cd client && npm run dev  -> http://localhost:5173
```

> `.env`, `vendor/`, and `client/node_modules/` are intentionally untracked, so `cp .env.example .env` + `make install` + `make client-install` are **required** on every new machine тАФ a bare clone will not boot until you run them.

Once up:

| Service  | URL / address | Notes |
|----------|---------------|-------|
| Frontend | http://localhost:5173 | **Use this** тАФ the Vite dev server. It proxies `/api` and `/sanctum` to the backend. |
| Backend  | http://localhost:8080 | nginx тЖТ PHP-FPM. You normally hit it through the frontend, not directly. |
| MySQL    | `localhost:3306` | user `cortex`, password `cortex`, db `cortex_lite` |
| Redis    | `localhost:6379` | |

### Daily use

```bash
make up                 # start everything
make client-dev         # (separate terminal) start the React dev server
make down               # stop everything
```

Common tasks: `make test`, `make shell`, `make logs`, `make artisan CMD="route:list"`, `make composer CMD="require ..."`, `make fresh` (drop + re-migrate + re-seed the DB).

### Environment variables тАФ required vs optional

The values in `.env.example` boot the app with **core features working** (auth, library, sessions, hardware DB, rule-based optimizer). The optimizer's AI prose and the external integrations only light up once you add the corresponding keys. The app is designed to degrade gracefully тАФ a missing key disables that one feature, it does not crash the request.

| Feature | Env vars | Needed to boot? | Behaviour if blank |
|---------|----------|-----------------|--------------------|
| Core app | `APP_KEY` | Yes тАФ generated by `make install` | тАФ |
| Steam library import | `STEAM_API_KEY` ([get one](https://steamcommunity.com/dev/apikey)) | No | "Connect Steam" / sync disabled |
| AI explanations | `GEMINI_API_KEY` | No | Falls back to deterministic static prose (recommendations are **never** affected) |
| Metadata enrichment | `PCGAMINGWIKI_CONTACT_EMAIL` | No | Metadata badges stay `pending` |
| Premium checkout | `STRIPE_KEY`, `STRIPE_SECRET`, `STRIPE_PRICE_PREMIUM`, `STRIPE_WEBHOOK_SECRET` | No | Upgrade button returns a config error тАФ see below |

After editing `.env`, apply it with `make restart` (or `make artisan CMD="config:clear"`).

### Stripe setup (test mode) тАФ needs a separate CLI process

Stripe checkout and webhooks need a bit more than an env var because Stripe cannot reach `localhost`. This is a **portfolio demo, so run Stripe in test mode** тАФ real public flow, fake money, only test cards accepted.

**1. One-time dashboard setup** (at [dashboard.stripe.com](https://dashboard.stripe.com), toggle **Test mode** on):
- Grab your test API keys тЖТ put them in `.env` as `STRIPE_KEY` (publishable, `pk_test_тАж`) and `STRIPE_SECRET` (secret, `sk_test_тАж`).
- Create a **Product** with a **recurring price of MYR 20.00 / month**. Copy the Price ID (`price_тАж`) тЖТ `.env` as `STRIPE_PRICE_PREMIUM`.

**2. Install the [Stripe CLI](https://stripe.com/docs/stripe-cli)** and log in once:
```bash
stripe login
```

**3. Every time you want to test billing, run this in its own terminal and leave it open:**
```bash
stripe listen --forward-to localhost:8080/api/stripe/webhook
```
It prints a webhook signing secret (`whsec_тАж`). Copy it into `.env` as `STRIPE_WEBHOOK_SECRET`, then `make restart`. (`stripe listen` is the local tunnel for the *inbound* webhook leg only тАФ it is not needed once deployed; production uses a Dashboard-registered webhook endpoint instead.)

**4. Test the flow** тАФ either click "Upgrade to Premium" in the app and pay with test card `4242 4242 4242 4242` (any future expiry, any CVC), **or** fire an event manually:
```bash
stripe trigger customer.subscription.created
```
Watch the `stripe listen` terminal report a `200` тАФ that confirms signature verification and the `is_premium` sync worked.

### Running without make

Every shortcut is a thin `docker compose` wrapper. Equivalents:

| make | raw command |
|------|-------------|
| `make up` | `docker compose up -d` |
| `make down` | `docker compose down` |
| `make install` | `docker compose exec app composer install && docker compose exec app php artisan key:generate --force && docker compose exec app php artisan migrate` |
| `make artisan CMD="db:seed"` | `docker compose exec app php artisan db:seed` |
| `make test` | `docker compose exec app php artisan test` |
| `make client-install` | `cd client && npm install` |
| `make client-dev` | `cd client && npm run dev` |

## Test flow тАФ verify everything end to end

After setup, sign in with the seeded demo account (see [Evaluator quick-start](#evaluator-quick-start)) and walk this path at http://localhost:5173:

1. **Auth** тАФ Register a new account (or log in as the demo user). Confirm you land on the dashboard; refresh to confirm the session persists (Sanctum cookie auth).
2. **Library** тАФ Go to `/library`, create a game manually, edit it, filter/search/sort, then delete it. *(Steam import needs `STEAM_API_KEY` тАФ with it set, use the dashboard "Connect Steam" panel and `Sync`.)*
3. **Sessions** тАФ Start a play session from a game, watch the persistent active-session banner, then end it. Check `/history` groups the session under its game.
4. **Hardware** тАФ On `/hardware`, use the GPU/CPU typeahead. Your picks are saved to a shared profile (localStorage) the optimizer reuses.
5. **Optimizer (forward)** тАФ On `/optimizer`, pick a game + GPU/CPU/RAM + a goal, run a recommendation. Results render as an in-game settings panel. With `GEMINI_API_KEY` set you get AI prose; without it, a static explanation тАФ recommendations are identical either way.
6. **Optimizer (reverse)** тАФ Toggle reverse mode, enter your current in-game settings, and confirm the diff view flags suboptimal values.
7. **Free-tier quota** тАФ As a free user, run recommendations until the 30-day quota (3 recommend / 5 reverse) is hit; confirm the `402` upgrade prompt appears.
8. **Billing** тАФ Follow the [Stripe setup](#stripe-setup-test-mode--needs-a-separate-cli-process) above, upgrade with test card `4242 4242 4242 4242`, and confirm the account flips to premium and quotas lift.

## Sprint changelog

- **Sprint 0 тАФ Setup, Docker, CI.** Laravel 13 + React 19 scaffolded. Six-service Docker Compose stack (app, nginx, mysql, redis, scheduler, queue). Multi-stage prod Dockerfile for the React тЖТ nginx image. GitHub Actions CI running PHPUnit on every push. Four docs (`ARCHITECTURE.md`, `DECISIONS.md`, `TROUBLESHOOTING.md`, this README) live under `docs/`.
- **Phase 1** тАФ Sanctum SPA auth (register/login/logout/me), password reset, email verification, throttled login with Retry-After, delete-account with Cashier subscription teardown. 34 feature + 6 unit tests (grew from the planned 23+6 as scope expanded with CashierInstallTest, CsrfTest, and expanded LoginTest coverage). Cashier installed early; no live Stripe surface yet.

- **Sprint 2 - Games library CRUD.** Added the user-scoped `games` table, manual CRUD API, IDOR-safe 404 behavior, wildcard-safe search, and a protected React `/library` page with filters, sorting, pagination, create/edit modal flow, and guarded delete. Steam OpenID/import remains the next Phase 2 sub-phase.
- **Sprint 2b - Steam OpenID + Web API integration.** Added Steam account linking via OpenID and direct SteamID64 fallback, transactional `/api/steam/sync`, nightly `steam:sync-all`, Redis-backed Steam response caching, and a dashboard Steam panel with connect/sync actions plus targeted privacy-settings guidance for the "Profile" and "Game Details" toggles.
- **Sprint 3 - Session tracking.** Added the play-session lifecycle (start/end/active/history), race-safe one-active-per-user invariant, manual-only playtime aggregation, persistent React active-session banner, and `/history` page grouped by game.
- **Sprint 4a - Hardware tier database.** Hand-curated `gpus.json` (~60 rows) and `cpus.json` (~40 rows) seeded via idempotent Laravel seeders. Tier is derived by pure PHP classifiers using absolute benchmark thresholds (GPU G3D Mark and CPU single-thread PassMark), making the classifier the single source of truth for boundary rules. Two auth-gated typeahead endpoints (`GET /api/hardware/gpus`, `GET /api/hardware/cpus`) power a reusable React `HardwareAutocomplete` component and a `/hardware` demo page that also surfaces browser-detected CPU cores, device memory, and WebGPU adapter presence with an explicit note that browsers cannot identify the GPU model.
- **Sprint 4b - PCGamingWiki metadata pipeline.** Added the `game_metadata` table, `PcGamingWikiClient`, Redis token-bucket limiter, AppID-only 7-day metadata cache, scheduled `games:enrich-metadata` command, and library-row metadata status badge. Pending Steam imports are enriched through PCGamingWiki Cargo and flipped to `ok` or `missing` for Phase 5's recommender masking.
- **Sprint 5 - AI optimizer backend.** Added deterministic forward recommendations, reverse-mode settings diffs, and AI explanations via `gemini-3.5-flash`. Gemini prose is cached in Redis by deterministic inputs and falls back to static strings so the LLM can never fail or alter a recommendation.
- **Sprint 5 - Stripe premium gating.** Added rolling 30-day free-tier quotas (3 recommendations / 5 reverse-mode calls), usage counters, RM20/mo Cortex Premium checkout, and signature-verified Stripe webhooks that sync `is_premium` from Cashier subscription state.
- **Sprint 5 - Optimizer UI.** New `/optimizer` page wires the recommendation and reverse-mode endpoints тАФ game picker, shared GPU/CPU/RAM profile (localStorage), goal selector, forward/reverse toggle, results styled as an in-game video-settings panel with quality notch meters, diff view, Gemini explanations, free-tier usage counters, and quota-402 upgrade flow.

- **Sprint 6 - AWS deployment artifacts.** Added the production PHP image with Parameter Store secret injection, RDS-oriented production compose topology, AWS Console runbook, ECR/EC2 helper scripts, native-agent telemetry contract, and a nightly reseeded evaluator demo account.
